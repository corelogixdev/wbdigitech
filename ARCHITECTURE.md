# WB DigiTech Laravel Project Architecture

## Overview

This is a Laravel application built with the standard Laravel MVC architecture:

- **Model**: `app/Models`
- **View**: `resources/views`
- **Controller**: `app/Http/Controllers`
- **Routes**: `routes/web.php`, `routes/auth.php`
- **Middleware**: `app/Http/Middleware`
- **Database schema**: `database/migrations`
- **Asset tooling**: `package.json`, `vite.config.js`, `resources/css`, `resources/js`

The application uses server-side rendering with Blade templates and a PHP backend. It is not a separate frontend SPA; instead, UI pages are generated by Laravel views and enhanced with CSS/JS via Vite.

---

## Folder Structure and Purpose

### Root folders

- `app/` - application core logic
- `bootstrap/` - framework bootstrap files
- `config/` - configuration files for database, app, cache, mail, etc.
- `database/` - migrations and seeders
- `public/` - document root for web server and built assets
- `resources/` - Blade views, styles, scripts, and frontend templates
- `routes/` - route definitions for web and auth
- `storage/` - compiled views, logs, cache, and runtime storage
- `tests/` - test cases
- `vendor/` - installed Composer dependencies

### `app/`

- `app/Http/Controllers/` - controllers that handle requests and return views or actions
- `app/Http/Middleware/` - middleware that filters requests before controllers run
- `app/Http/Requests/` - request validation classes (if used)
- `app/Models/` - Eloquent models representing database tables
- `app/View/Components/` - Blade component classes used to build layouts
- `app/Providers/` - service providers and application bootstrapping

### `config/`

Contains app configuration, including:

- `config/database.php` - database connection settings
- `config/app.php` - app settings and providers
- `config/session.php`, `config/cache.php`, etc.

### `database/`

- `database/migrations/` - database schema definitions
- `database/seeders/` - seed data loaders

### `public/`

- `public/index.php` - Laravel front controller
- `public/build/` - compiled Vite assets
- `public/css/`, `public/js/` - public static files output or assets

### `resources/`

- `resources/views/` - Blade templates and page views
- `resources/css/` - CSS source files
- `resources/js/` - JS source files
- `resources/sass/` - Sass source files

### `routes/`

- `routes/web.php` - public and admin route definitions
- `routes/auth.php` - authentication and password/reset routes
- `routes/console.php` - console commands (standard Laravel)

## Frontend vs Backend

### Frontend

The frontend is implemented as Laravel Blade views and static pages.

Primary frontend folders:

- `resources/views/`
  - `pages/` - public page templates
  - `layouts/` - layout templates
  - `components/` - reusable Blade components
- `resources/css/` - style files
- `resources/js/` - frontend scripts
- `public/` - compiled/public assets

There are also many static service pages routed directly to Blade views from `routes/web.php`, such as:

- `pages/services/website/`
- `pages/services/google_ads/`
- `pages/services/seo/`
- `pages/services/social_media_marketing/`
- `pages/services/mobile_application/`

### Backend

The backend is Laravel PHP code that handles application state, database access, and admin management.

Primary backend folders:

- `app/Http/Controllers/`
- `app/Models/`
- `database/migrations/`
- `database/seeders/`
- `config/`
- `routes/`

Key backend controllers include:

- `HomeController` - homepage, about page, and public dashboard data
- `BlogController` - blog CRUD and public blog pages
- `ContactController` - contact form submission + admin message management
- `SeoRequestController` - SEO request handling
- `TeamController`, `CustomerController`, `IndustryController`, `CareerController`, `PortfolioController`, `ServiceController`, `UserController` - admin management for those entities
- `DashboardController` - admin dashboard

---

## Middleware

Primary middleware folder:

- `app/Http/Middleware/`
  - `RoleMiddleware.php`

`RoleMiddleware` performs:

- authentication check
- authorization based on user role
- aborts with `403 Unauthorized` if the user is not logged in or does not match allowed roles

Standard Laravel auth middleware is also used via `routes/auth.php` and likely registered in `app/Http/Kernel.php`.

## Where to make changes

To modify application behavior:

- Edit backend logic in `app/Http/Controllers/`
- Edit database models and fillable fields in `app/Models/`
- Add or adjust routes in `routes/web.php` and `routes/auth.php`
- Change request validation in `app/Http/Requests/` or controller `validate(...)`
- Change middleware in `app/Http/Middleware/`
- Change frontend output in `resources/views/`
- Change CSS/JS assets in `resources/css/`, `resources/js/`, `resources/sass/`

To rebuild frontend assets after changes:

```bash
npm install
npm run dev
# or
npm run build
```

To refresh Laravel config and run database migrations:

```bash
php artisan config:clear
php artisan cache:clear
php artisan migrate
```
